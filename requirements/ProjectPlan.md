# План Проєкту (Лекція 5 — Управління)

## Мета документа
Зафіксувати ключові ризики та узгодити базовий план управління індивідуальним проєктом "Фітнес Тренер AI" (React + Firebase + Gemini AI). Документ доповнює SRS_v1.md та Tech_Zavdannya.md і показує, що ризики передбачені ще до завершення впровадження.

## План управління
### Ролі та відповідальності (Solo-розробник)
- **Technical Lead / Архітектор:** визначення стеку (React, Firebase, Gemini), дизайн API та сервісів.
- **Front-end Engineer:** реалізація UI (App.tsx, ExerciseCard, SystemStatusBar) з дотриманням золото-чорної теми.
- **Back-end/Firebase Engineer:** моделювання даних, безпека Firestore, офлайн-синхронізація.
- **ML/AI Integrator:** промпт-інжиніринг, генерація планів, адаптивні тренування.
- **QA Engineer:** написання тест-кейсів, smoke-тест, валідація офлайн-сценаріїв та health-flow.

### Етапи розробки
1. **Ініціація (T0–T1):** збір вимог (SRS_v1, Tech_Zavdannya), підготовка архітектури й макетів.
2. **Розробка ядра (T1–T3):** автентифікація, профіль користувача, генерація базового плану, UI тренувань.
3. **AI та адаптація (T2–T4 перекриття):** інтеграція Gemini, health-profile, інструменти адаптації вправ.
4. **Офлайн та DevEx (T3–T5):** PWA, Service Worker, offline queue, SystemStatusBar.
5. **Тестування та стабілізація (T4–T5):** регресії, smoke-тест, оновлення документації (backlog, TestCases).

### Моніторинг та контроль
- Щотижневий огляд прогресу за канбан-дошкою (TODO/In progress/Done).
- Метрики: кількість завершених user stories, результати smoke-тестів, помилки синхронізації.
- Управління змінами: важливі правки логуються в docs/* та requirements/* (цей документ, backlog, TestCases).
